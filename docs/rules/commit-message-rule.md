# Git コミットメッセージ運用ルール

本リポジトリでは、
**「変更の性質が一目で分かること」** を目的として、
prefix を最小限に絞ったコミットメッセージ運用を行う。

形式は以下に統一する。

---

## 使用する prefix 一覧（最小構成）

### feat

**新しい機能の追加**

- ユーザー価値が増える変更
- 新しい処理・オプション・フローの追加

例：feat: 未処理日報を最大5件まで処理する機能を追加

---

### fix

**不具合修正**

- 想定外の挙動を正しい状態に戻す
- バグ・取りこぼし・条件ミスの修正

例：fix: Notion blocks取得時のページング漏れを修正

---

### change

**仕様変更（挙動や結果が変わる）**

- バグではないが、動作・出力・ルールが変わる
- 表示位置、出力先、フォーマット変更など

例：change: フィードバック出力先をプロパティから本文に変更

※ `refactor` と迷った場合、
**ユーザーが見て「変わった」と感じるなら change を使う**

---

### refactor

**外部仕様を変えない内部整理**

- 処理結果・挙動・出力は同じ
- 構造整理、責務分離、命名改善、関数分割など

例：refactor: Notion取得処理をhelpersに分離

---

### docs

**人間向けドキュメントの変更**

- README、設計書、運用ルール
- コメント整備（説明目的）

例：docs: バッチ処理フローの設計書を追加

※ ツール挙動を制御するファイル（AGENT.md 等）は docs ではなく `chore`

---

### test

**テスト関連の変更**

- テスト追加・修正・削除
- 期待値やモックの調整

例：test: runBatchの本文抽出テストを追加

---

### chore

**運用・設定・雑務（アプリ挙動は基本変えない）**

- ツール設定、CI補助、開発体験改善
- 依存関係更新、設定ファイル追加

例：chore: CodexのPR自動レビューを日本語化

---

## prefix 判定ルール（迷ったとき）

1. **結果・表示・挙動が変わるか**
   - YES → `change`
   - NO → 次へ

2. **新しい機能か**
   - YES → `feat`
   - NO → 次へ

3. **バグ修正か**
   - YES → `fix`
   - NO → 次へ

4. **内部整理のみか**
   - YES → `refactor`
   - NO → 次へ

5. **説明用ドキュメントか**
   - YES → `docs`
   - NO → 次へ

6. **それ以外（設定・運用）**
   - → `chore`

---

## 採用しない prefix（意図的に使わない）

- perf
- build
- ci
- style
- revert（git revert 実行時は自動生成に任せる）

理由：

- prefix が増えすぎると判断コストが上がる
- 運用が形骸化しやすくなるため

---

## このルールの方針

- 正確さより **一貫性**
- 迷ったら **change / chore**
- prefix は「履歴を追いやすくするための補助情報」

本ルールは、
個人開発〜小規模プロジェクトでの
**長期的に破綻しない運用**を目的としている。
